Question 1 of 1

Design a real-time agent-streaming backend that survives disconnects.

Your answer

I opened with functional requirements (multi-turn chat, tool calls visible, graceful disconnects) and non-functional (p95 first-token < 1.5s, reconnect window ~60s). Then the high-level: a FastAPI transport layer over a LangGraph state machine with a Postgres checkpointer, each turn running a supervisor that routes to phase agents. Streaming is SSE with monotonic cursors; the client transport holds the last cursor and reconnects with Last-Event-ID. For scaling, the checkpointer lets us horizontally shard agent workers by conversation ID. For trade-offs, I picked SSE over WebSockets because unidirectional server → client is all I need, and chose a Postgres checkpointer over Redis for durability even though Redis would be faster.

Ideal answer

Same structural answer. Strongest responses also mention: (a) backpressure when the model outpaces the client — I'd use a bounded queue and drop intermediate 'thinking' events on overflow rather than token events, (b) a tool-approval interrupt flow using LangGraph's DeferredToolRequests so the turn pauses on the server without the client needing to know how, and (c) observability — tagging every SSE event with turn_id + cursor so you can reconstruct a failed session from logs.
